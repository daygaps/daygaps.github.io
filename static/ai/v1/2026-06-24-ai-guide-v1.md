# DayGaps Bridge: a guide for AI assistants (v1)

You have been handed a `daygaps_bridge:` payload and a folder of YAML files. This page tells you how to edit those files correctly. Read it before you change anything.

## What the bridge is

The DayGaps Bridge gives you a folder of YAML files that represent part of a person's DayGaps workspace: their projects, the tasks inside those projects, and how the tasks are grouped and scheduled. While the bridge is open, DayGaps is frozen on every device the person owns, and these files are the source of truth. You edit the files in place, and when the person closes the bridge, DayGaps reads your edits back, shows them a before-and-after diff, and they decide whether to apply it.

## The folder you were given

The folder holds one YAML file per project, named `<area>/p-<id>.yaml`. An "area" is the category a project lives in, and it is just the parent folder name.

Edit only the project files. You will see other files in the folder: `areas.yaml`, `inbox.yaml`, an `<area>/_tasks.yaml` for each area's loose tasks, and a `days/` folder. Leave all of them exactly as they are. In v1 the editing surface is project files only; the rest of the folder is part of the snapshot but off limits.

## A project file, field by field

A project file has two parts: a `project:` block of metadata, then a `tasks:` list.

```yaml
project:
  id: p-9d74be
  name: "Ideas for DayGaps 1.0"
  status: active
  purpose: |-
    **2026-06-22 11:21 PDT:** decided on the coral underline
  due_date: 2026-07-01
  last_edited: 2026-06-24T03:05:21Z
  me:
    sort_index: 4000

tasks:
  - id: t-3cc949
    title: "Make links in task titles clickable"
    done: true

  - header: "Visual polish"

  - id: t-c882b0
    title: "Bigger project titles on area pages"
    done: false
    due_date: 2026-06-30
    me:
      date: 2026-06-23
      gap: "09:00"
    subtasks:
      - id: t-77af01
        title: "Pick the heading weight"
        done: false
```

The `project:` block:

- `id`: required. Format `p-` followed by a short hash. It identifies the project. Never change it.
- `name`: required. The project's title. Wrap it in double quotes.
- `status`: required. It will read `active` for every project you edit. Leave it as `active`.
- `purpose`: optional. The project's notes, in markdown. See "The purpose field" below.
- `due_date`: optional. A project-level deadline, in `YYYY-MM-DD` form.
- `last_edited`: an informational timestamp. DayGaps does not read it back, so you do not need to update it.
- `me`: optional. The person's own view settings for this project: `pinned: true` and `sort_index: <whole number>`. Absent means not pinned and default order. You rarely need to touch this.

The `tasks:` list is ordered. The order of the items in the file is the order the person sees in the app. Each item is either a task or a section header.

## Section headers, tasks, and subtasks

There are two kinds of item in a `tasks:` list.

A task looks like this:

```yaml
  - id: t-3cc949
    title: "Make links in task titles clickable"
    done: false
```

- `id`: optional. Format `t-` followed by a short hash. Every existing task already has one. Keep it unchanged. When you add a brand new task, leave `id` out and DayGaps assigns one for you.
- `title`: required. Wrap it in double quotes.
- `done`: `true` or `false`. If you leave it out, it defaults to `false`.
- `due_date`: optional task deadline, in `YYYY-MM-DD` form.
- `me`: optional schedule. See "The me block" below.
- `subtasks`: optional nested list. See below.

A section header looks like this:

```yaml
  - header: "Visual polish"
```

A header groups the tasks that follow it, until the next header or the end of the list. Headers have no id. Their position in the list is their identity, so to "rename" a header you edit its text in place, and to move a group you move the header and its tasks together.

Subtasks live nested under their parent task, inside a `subtasks:` list:

```yaml
  - id: t-c882b0
    title: "Bigger project titles on area pages"
    done: false
    subtasks:
      - id: t-77af01
        title: "Pick the heading weight"
        done: false
```

A subtask carries only three fields: `id` (optional, same rule as a task id), `title`, and `done`. Subtasks do not take a `me` block, a `due_date`, or further nesting. Keep them inside their parent's `subtasks:` list. This is the single most common way an edit goes wrong, so it has its own warning under "What not to do."

## The me block

The `me:` block on a task is the person's schedule for that task.

```yaml
    me:
      date: 2026-06-25
      gap: "09:00"
```

- `date`: optional. The day the task is scheduled for, in `YYYY-MM-DD` form.
- `gap`: optional. A time of day in 24-hour `"HH:mm"` form, in quotes. It names the gap (the open block of time) in the person's day that the task should sit in.

You can supply `date`, `gap`, or both. To schedule a task, add a `me:` block with the fields you want. To unschedule it, remove the `me:` block (or the single field you want to clear); an absent `me:` block means the task has no date and no gap. The block is per person, and in v1 there is one person, so read it as "their schedule for this task." Never put a `me:` block on a subtask.

## The purpose field

`purpose:` holds the project's notes as markdown, written as a YAML block scalar (the `|-` marker, with every line of the note indented under the key):

```yaml
  purpose: |-
    **2026-06-22 11:21 PDT:** decided on the coral underline
    Still need to confirm the dark-mode value.
```

The app records dated notes as paragraphs that begin with a bold timestamp, for example `**2026-06-22 11:21 PDT:** ...`. If you add a dated note, follow that shape. Otherwise edit the prose as needed. Keep everything inside the one `purpose:` block; do not split notes into new top-level keys.

## Editing and the lifecycle

The bridge has three stages: open (right now), closing, and closed. You only ever act while it is open.

Editing means changing the files on disk and saving them. There is no separate "commit" or "send" step. Saving the files is the whole job.

When you finish, save every file you changed and tell the person you are done. They then close the bridge in DayGaps. DayGaps compares your saved files against the snapshot it took when the bridge opened, shows the person a project-by-project before-and-after diff, and they choose one of:

- Apply: your edits are written into DayGaps.
- Discard: your edits are thrown away and nothing is written.

Nothing reaches DayGaps until the person applies. Only the files you actually changed are examined; files you left untouched are skipped.

## If the bridge closes while you are working

The person can close the bridge at any moment, including while you are partway through editing. When that happens, DayGaps removes the YAML files from the folder.

Treat the folder as live. If it becomes empty, or the files you were editing have disappeared, the bridge has closed. Stop, tell the person it looks like the bridge closed, and do not recreate the files. Files you recreate are not part of any transaction and can collide with a later session. Do not retry in a loop.

## The person's other devices are frozen

While the bridge is open, DayGaps is frozen on all of the person's other devices. You cannot ask them to make a change on their phone or on another computer, and nothing on another device can be edited until the bridge closes. Work only with the files in front of you.

## What not to do

- Do not change any `id:` value. Ids are how DayGaps tracks a task or project across the person's devices and across this edit. A changed id reads as "delete the old one, create a new one," which loses the item's identity and its history.
- Do not add or remove files. The bridge applies changes one file at a time against the opening snapshot. A file you add has nothing to match and is rejected. A file you delete is ignored, not read as "delete this project." To empty a project, clear its `tasks:` list rather than deleting its file.
- Do not add new top-level keys to a file. DayGaps ignores keys it does not recognize, so anything you invent is dropped the next time the file is read. Use the fields documented here.
- Do not flatten subtasks. Keep each subtask nested inside its parent task's `subtasks:` list. A flat list that uses a `parent_id` (or similar) field breaks the parent-child link, and the subtasks become loose top-level tasks.
- Do not edit files outside the bridge folder, and do not edit the `daygaps_bridge:` payload the person pasted to you. Neither is part of the transaction.

## Worked example

Start from this project file, `business/p-9d74be.yaml`:

```yaml
project:
  id: p-9d74be
  name: "Ideas for DayGaps 1.0"
  status: active
  last_edited: 2026-06-24T03:05:21Z

tasks:
  - id: t-3cc949
    title: "Make links in task titles clickable"
    done: true

  - id: t-c882b0
    title: "Project titles on area pages"
    done: false
    me:
      date: 2026-06-23
```

A reasonable set of edits: add a section header, move the second task under it, sharpen its title, reschedule it to the 25th in the 09:00 gap, and add a new follow-up task with one subtask.

```yaml
project:
  id: p-9d74be
  name: "Ideas for DayGaps 1.0"
  status: active
  last_edited: 2026-06-24T03:05:21Z

tasks:
  - id: t-3cc949
    title: "Make links in task titles clickable"
    done: true

  - header: "Visual polish"

  - id: t-c882b0
    title: "Make project titles on area pages bigger and bolder"
    done: false
    me:
      date: 2026-06-25
      gap: "09:00"

  - title: "Audit heading weights across every screen"
    done: false
    subtasks:
      - title: "List each screen that shows a heading"
        done: false
```

What round-trips here: the two kept tasks keep their ids; the header lands as a group above the task that follows it; the reschedule rides along in the `me:` block; the new task and its subtask have no id, so DayGaps assigns fresh ones when the person applies. Re-open the bridge afterward and you would see the new task carrying its assigned `t-` id, the header still in place, and the schedule preserved.

What would have gone wrong: changing `t-c882b0` to a different id would drop that task and recreate it as a stranger. Lifting the subtask out to the top level with a `parent_id` field would orphan it. Adding a `tags:` key to the file would vanish on the next save.
